# AetherOS — Backend Development Document

This document specifies the backend that must be built to make every currently frontend-only feature production-real. The frontend is intentionally ahead of the backend so this doc can be executed against a stable UI contract.

Stack: TanStack Start (Cloudflare Worker runtime) · Supabase (Postgres + Auth + RLS) · Lovable AI Gateway (AI SDK) · optional pg_cron + external workers for long-running orchestration.

## 2. Database Schema (new tables)

All tables live in `public`, always with `GRANT` + RLS. Every mutating table gets `created_at`, `updated_at`, `org_id`, and `user_id` where relevant.

### 2.1 Multi-tenancy

```sql
create table public.organizations (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  slug text unique not null,
  created_at timestamptz not null default now()
);

create table public.org_members (
  org_id uuid references public.organizations(id) on delete cascade,
  user_id uuid references auth.users(id) on delete cascade,
  role app_role not null default 'user',
  primary key (org_id, user_id)
);
```

Add `org_id uuid not null` to `projects`, `agent_runs`, `artifacts`, `approvals`, and every new table below. RLS policies must scope by `org_id` via a `is_org_member(org_id, auth.uid())` security-definer function.

**Every** `create table` above must be followed in the same migration by:

```sql
grant select, insert, update, delete on public.<table> to authenticated;
grant all on public.<table> to service_role;
alter table public.<table> enable row level security;
create policy <table>_org_rw on public.<table> for all to authenticated
  using (public.is_org_member(org_id, auth.uid()))
  with check (public.is_org_member(org_id, auth.uid()));
```

Global catalog tables (`frameworks`, `controls`, `marketplace_items`, catalog `agents`, catalog `kg_nodes`) additionally grant `select` to `anon`/`authenticated` with a read-only policy.

## 3. Server Functions (TanStack `createServerFn`)

App-internal calls only — never Supabase Edge Functions.

Naming convention: `src/lib/<domain>.functions.ts` with `.middleware([requireSupabaseAuth])`.

Required functions (grouped):

- `projects.functions.ts` — existing + `cloneProject`, `archiveProject`, `getProjectVersions`.
- `blueprints.functions.ts` — `snapshotBlueprint`, `listVersions`, `diffVersions`, `restoreVersion`.
- `comments.functions.ts` — `addComment`, `listComments(projectId, artifactKind)`, `deleteComment`.
- `audit.functions.ts` — `listAudit(projectId?, orgId, filters)`, `exportAuditCsv`.
- `approvals.functions.ts` — extend existing: `listPendingApprovals(orgId)`, `decideApproval` (also writes audit event + triggers next orchestrator step).
- `kg.functions.ts` — `searchNodes(q)`, `getNode(id)`, `neighbors(id)`, admin `upsertNode` / `upsertEdge`.
- `simulation.functions.ts` — `startSimulation(projectId, scenarioKey, params)` → returns `simulationId`; `getSimulation`. Server fn enqueues; a worker fulfills.
- `governance.functions.ts` — `listPolicies`, `evaluatePolicy(projectId, policyId)`, `bulkEvaluate(projectId)`.
- `compliance.functions.ts` — `evaluateFramework(projectId, frameworkId)`, `listEvaluations(projectId)`.
- `finops.functions.ts` — `getSpend(range, groupBy)`, `getForecast(projectId)`, `listOptimizations`.
- `marketplace.functions.ts` — `listItems(filter)`, `installItem(projectId, itemId)`.
- `operations.functions.ts` — `listIncidents`, `listOptimizations`, `acknowledgeIncident`.
- `deployments.functions.ts` — `createDeployment`, `getDeployment`, `rerun`, `abort` (permission-guarded).
- `organization.functions.ts` — `listMembers`, `invite`, `updateRole`, `removeMember`, org CRUD.

Rules:
- Read `process.env.*` **inside** handlers only.
- Never top-level import `@/integrations/supabase/client.server` in a `.functions.ts` file; load it inside the handler with `await import(...)` and only after verifying the caller's role via `context.supabase` + `has_role`.
- Zod-validate every input; return DTOs only.

## 4. Orchestrator (multi-agent pipeline)

The current implementation streams via `/api/chat` (great for real-time UX) but is not durable. Replace with a durable orchestrator:

**Design:**
1. `POST /api/chat` becomes a **client bridge** that opens an SSE connection to a running orchestrator (in-worker for short runs, external for long ones).
2. Orchestrator state lives in a single `orchestrations` row with a JSON graph of agent steps + status.
3. Each step is a queued job: agent → tool call → structured output → persist artifact + audit event → decide next step.
4. Approval gates set step status to `waiting_approval` and pause the run; `decideApproval` resumes it.
5. Clients reconnect by ID and receive the tail of the event log plus live updates.

Recommended implementation:
- **Queue:** Cloudflare Queues or Supabase `pg_cron` + row-based work claiming. For >1 min steps, offload to an external Node worker.
- **Streaming:** keep AI SDK `streamText` inside each worker step so tokens can be forwarded via SSE.
- **Retries:** exponential backoff per step; poison queue on 3 failures.

**Agents table drives the runtime.** The 12+ agent metadata currently in `src/lib/agents.ts` becomes the seed for the `agents` table. New agents are added via migration (or admin UI) without redeploy.

Layer taxonomy (must match UI):
`planning · discovery · architecture · governance · generation · validation · operations`.

Each agent record specifies: tool schema (Zod → JSONSchema), system prompt, model preference, `needsApproval`, and downstream dependencies.

## 5. Real-time

Supabase Realtime channels (server publishes via row inserts):

- `projects:<projectId>` — `agent_runs`, `artifacts`, `approvals`, `audit_events`.
- `orgs:<orgId>:activity` — org-wide activity feed.
- `orgs:<orgId>:approvals` — approval center live counter.

Client subscribes in the relevant route; `useQuery` invalidation on channel messages.

## 6. Ingestion Jobs

Scheduled via `pg_cron` calling a public server route (`/api/public/jobs/*`) with an HMAC-signed body:

- Cost ingest (hourly): pull provider billing → `cost_facts`.
- Compliance evaluate (nightly): re-score all active projects.
- Well-Architected review (nightly): run reviewer agent, upsert `wa_reviews`.
- Optimization scan (nightly): produce right-sizing suggestions.
- Knowledge graph enrichment (nightly): embed new artifacts + link.

All jobs are idempotent, take a `since` cursor, and write an `audit_events` entry on completion.

## 7. Integrations (MCP + connectors)

- Use `ai-sdk-mcp-client` knowledge for runtime MCP.

- Connectors of interest: AWS (Cost Explorer, CloudFormation, IAM), Azure, GCP, GitHub (repos + PRs), Jira, Slack, PagerDuty, Datadog, Snyk.
- Each connector exposes a small tool surface consumed by relevant agents (FinOps → cost tools, Reliability → PagerDuty, IaC → GitHub).

## 8. Security

- RLS on **every** table. No exceptions. Test with an anon token in CI.
- `SUPABASE_SERVICE_ROLE_KEY` only inside `.server.ts` helpers behind role checks.
- Rate-limit `/api/chat` per user (token bucket in KV) — the AI Gateway 429/402 must surface in UI as we already do.
- OPA/Rego bundles signed and versioned. Policy changes require a governance approval.
- Audit log is append-only; enforce with a trigger that raises on `UPDATE`/`DELETE`.

## 10. Build Order (recommended)

1. **Multi-tenant foundation** — `organizations`, `org_members`, `is_org_member`, backfill `org_id`.
2. **Audit + comments + blueprint versions** — remove localStorage dependency.
3. **Durable orchestrator** — migrate `/api/chat` to job-backed runs with resume.
4. **Knowledge graph** — seed + pgvector search + agent context.
5. **Governance + compliance eval jobs** — replace mocked pages.
6. **FinOps ingestion** — real cost data flowing.
7. **Deployments** — Terraform runner + policy gates + audit.
8. **Marketplace + integrations** — install flow + MCP connectors.
9. **Operations** — telemetry ingest + optimization agent.

Each step is independently shippable. UI is already contract-frozen — build to what the pages consume.
